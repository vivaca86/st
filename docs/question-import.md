## 확인된 결론 (다시 시도하지 말 것)

### 1. 텍스트 레이어 파싱은 죽은 길이다 — 수율 2%

PDF에는 GoodNotes가 만든 OCR 텍스트 레이어가 있고 **한글 지문은 깨끗하게
추출된다.** 그래서 텍스트로 파싱하고 싶어지지만, 실측 결과 쓸 수 없다.

`전기자기/12.전자계.goodnotes`로 검증했다. `정답` 마커 42개 중 온전히
파싱된 문제는 **1개**였다. 나머지는:

- **수식이 파괴된다.** `1 J= 2TE,E Eid 3 2TE J=4 TE, E 4 f= 4ㅠ2E`
- **①②③④가 비결정적으로 깨진다.** 같은 `0`이 ①·②·③ 모두를 뜻하고,
  같은 ④가 `Ф`·`®`·`4`·`G`로 제각각 나온다. 역산 불가능.
- **읽기 순서가 뒤엉킨다.** 2단 조판 + 해설이 선택지 사이에 끼어들어,
  정규식 파서가 뒤 문제 3개를 선택지 하나로 통째로 삼켰다.
- **OCR 오타.** `자성체`를 `자상체`로 읽었다.

가장 위험한 점: 파서가 **구조는 멀쩡한데 내용이 쓰레기인 JSON을 자신만만하게
뱉는다.** 이대로 돌리면 그럴듯한 오염 문제 수천 개가 DB에 들어간다.

시도 흔적: `scripts/import-goodnotes.mjs`가 이 접근이다. 아래 6번 참고.

### 2. 스캔 이미지는 완벽하게 읽힌다 — 이것이 유일한 길

같은 페이지를 이미지로 렌더링하면 지문·선택지·정답·해설이 전부 또렷하다.
`전기자기/12.전자계` 1페이지에서 4문제를 **4/4 정확**하게 옮겼고,
텍스트 파서가 틀린 것(위 1번의 모든 사례)을 전부 바로잡았다.

### 3. 정답은 인쇄되어 있다 — 추론하지 말 것

**문제를 풀어서 정답을 맞히려 하지 마라.** OCR로 깨진 수식을 보고 정답을
추론하면 그럴듯하지만 틀린 답이 섞이고, 그게 "검수 완료"로 표시되어
사용자가 틀린 것을 외우게 된다.

정답은 원본에 인쇄되어 있으므로 **판독**만 하면 된다:

- **이론 노트 / 기출문제**: 문제마다 `정답 ③`이 인라인으로 붙어 있고
  `정답분석` 박스에 해설도 있다.
- **문제 폴더 / 회차 시험**: 페이지 하단에 `정답 17④ 18② 19④ 20② 21②`
  형태의 정답표 띠가 있다.

정답표 판독은 검증되었다. 2025년 1회에서 이미지로 읽은 9개가 사람이 검수한
`data/seed-questions.json`의 정답과 **9/9 일치**했다.

### 4. 정답표 판독은 언제든 재검증할 수 있다

`data/seed-questions.json`의 100개는 2025년 1회의 사람 검수 정답이다.
판독 로직이 의심스러우면 2025년 1회로 대조하면 100% 채점된다.

### 5. 검수되지 않은 문제는 출제되지 않는다

`app/api/exams/route.ts`는 `review_status = 'verified' AND answer_index IS NOT NULL`
인 문제만 출제한다. 정답이 불확실하면 `needs_review`로 넣어라. 버리지 말 것.
검수 화면(`components/ImportanceReview.tsx`)에서 사람이 확인하면 풀에 들어간다.

### 6. `scripts/import-goodnotes.mjs`를 그냥 실행하지 말 것 — 지뢰

기본 모드가 `curated`이고, 하드코딩된 `CURATED` 배열(**25개**)로
`data/seed-questions.json`(**100개**)을 **덮어쓴다.** 75문제가 날아간다.
또한 2.5GB 원본을 `JSON.parse(fs.readFileSync(...))`로 읽으려 하는데
Node의 문자열 한계가 0.5GB라 어차피 즉사한다.

### 7. DB는 시드 파일에서 채워진다

`db/runtime.ts`가 워커 시작 시 `data/seed-questions.json`에서 시딩한다.
`INSERT INTO questions`는 저장소 전체에서 이 한 곳뿐이다.
**문제를 늘린다 = 시드 JSON을 키워서 배포한다.**

### 8. 배포 방법은 아직 미해결

이 앱은 사용자의 Cloudflare 계정에 없다(D1은 `team-workspace` 하나뿐,
R2는 비활성). `.openai/hosting.json`의 `appgprj_6a5728965c748191a394b3ab82f35075`로
OpenAI 앱 호스팅이 굴린다. wrangler 설정도 배포 스크립트도 CI도 없다.
**앱 URL을 확보해서 `/api/dashboard`로 실제 DB 개수를 확인해야 한다.**

### 출력 형식

`data/seed-questions.json`과 동일한 스키마를 따를 것:

```json
{
  "subject": "전기자기학",
  "source_doc": "전기자기/12.전자계.goodnotes",
  "source_page": 1,
  "source_no": 1,
  "duplicate_group": "전자계-q001",
  "stem": "변위전류의 개념 도입은 다음 중 누구의 기여에 의한 것인가?",
  "choices": ["패러데이(Faraday)", "렌츠(Lenz)", "맥스웰(maxwell)", "로렌츠(Lorentz)"],
  "answer": 3,
  "explanation": "...",
  "confidence": 0.99,
  "needs_review": false
}
```

- `answer`는 **1부터 시작**한다 (①=1). `db/runtime.ts`의 `parseAnswerIndex` 참고.
- `subject`는 `lib/subjects.ts`의 5개 과목명과 정확히 일치해야 한다.
- 수식이 확실하지 않으면 `needs_review: true`, `confidence`를 낮게.
- `confidence`는 측정값이 아니라 판단값이다. 기존 시드는 100개 중 88개가
  0.99인데 이는 사람이 적어 넣은 숫자다.

### 수식 표기 규칙 (중요)

**분수는 반드시 분수로 보이게 적을 것.** 옮겨 적기 편하다고 `(Vm/ωC)sin(ωt+π/2)`
처럼 한 줄로 적으면 안 된다. 문제를 푸는 사람이 읽기 어렵다.

`components/FormulaText.tsx`가 두 가지를 스택 분수로 렌더한다:

| 표기 | 결과 |
|---|---|
| `\frac{a}{b}` | 항상 분수로 쌓임. **이걸 쓸 것** |
| `a ÷ b`, `a/b` | a와 b가 "atom"일 때만 쌓임 |

자동 규칙(`a/b`)은 글자 클래스가 `A-Za-zΑ-Ωα-ωℓ`뿐이라 **`∂`가 빠져 있다.**
따라서 `∂D/∂t`는 분수로 안 쌓이고 한 줄로 나온다. 편미분·복잡한 식은
**반드시 `\frac{}{}`를 명시**할 것. JSON 안에서는 `"\\frac{∂D}{∂t}"`로 이스케이프.

검증됨: `\frac{∂Nφ}{∂t}`, `\frac{Vₘ}{ωC} sin(ωt + \frac{π}{2})` 모두 분자/분모가
12~14px 어긋난 스택 분수로 렌더되고, 스크린리더에는 "∂Nφ 나누기 ∂t"로 읽힌다.

첨자는 유니코드를 쓴다: `ε₀`, `a²`, `10⁻⁷`, `Vₘ`, `[A/m²]`.
기존 시드 100개는 `÷`와 유니코드 첨자만 쓰고 `\frac`은 쓰지 않았다
(`−Q ÷ (2πa²) C/m²`). 두 표기 모두 렌더되므로 섞여도 무방하다.

### 그림 의존 문제

지문이 "그림에서…"처럼 도표를 참조하는 문제는 **앱이 문제 이미지를 지원하지
않으므로** 그대로 넣으면 풀 수 없는 문제가 된다. `needs_review: true`와
`review_reason`을 붙여 격리할 것. 비율을 계속 집계해서, 크면 R2 이미지 지원
(`assetManifest`)을 붙일지 판단한다.

## 우선순위

1. **이론 노트 68개** — 문제당 정답·해설이 인라인이라 가성비 최고
2. **기출문제 15개**
3. **문제 폴더 22개** — 2,082문제로 가장 크지만 정답표 판독 필요
4. **회차 2·3회** — 1회는 이미 반영됨

1번만 끝나도 100개 → 1,000개 이상이 된다. 전체를 다 하지 않아도 앱은 쓸만해진다.

## 비용 실측

- 페이지 1장 200 DPI = 약 4,100 토큰 / 150 DPI = 약 2,300 토큰
- 파일 1개(18p) = 약 10만 토큰
- 전체 1,389페이지 = 약 300만~570만 토큰

한 세션에 끝나지 않는다. 파일 단위로 끊어서 여러 세션에 나눠 처리할 것.
